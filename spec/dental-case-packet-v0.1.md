# Dental Case Packet Specification v0.1

Status: Draft  
Spec ID: `DCP-0.1`  
Canonical media type: `application/vnd.clinicbrain.dental-case-packet+json;version=0.1`  
Clinical status: Non-diagnostic; for dentist review only

## 1. Abstract

The Dental Case Packet is a JSON-based context format for representing de-identified dental records in a way that can be consumed by LLMs, agent systems, dental copilots, and clinical review workflows.

The specification defines a portable packet structure, validation rules, privacy requirements, compatibility expectations, and interoperability direction. It does not define diagnostic logic and does not define treatment planning logic.

## 2. Normative Language

The key words `MUST`, `MUST NOT`, `REQUIRED`, `SHOULD`, `SHOULD NOT`, and `MAY` are to be interpreted as described in RFC 2119.

## 3. Goals

- Standardize AI-ready dental context.
- Preserve references to source artifacts without embedding large files.
- Provide deterministic metadata, summaries, missing-information markers, and review questions.
- Make privacy and de-identification default behavior.
- Keep all outputs suitable for dentist review.
- Enable future mappings to FHIR, DICOM metadata profiles, and LLM connector formats.

## 4. Non-Goals

- Diagnosis.
- Treatment recommendations.
- Clinical accuracy claims.
- Replacement of dentist review.
- Pixel-level medical image interpretation.
- Training a foundation model.

## 5. Packet Object

A Dental Case Packet v0.1 document MUST be a JSON object.

It MUST include:

- `case_id`
- `created_at`
- `patient`
- `chief_complaint`
- `clinical_notes_summary`
- `treatment_plan_summary`
- `imaging`
- `ai_ready_context`
- `safety`

It SHOULD include future metadata fields:

- `packet_version`
- `schema_url`
- `source_manifest`
- `provenance`

These are not required in v0.1 because the reference implementation predates the formal spec-first layout.

## 7. Field Definitions

### 7.1 `case_id`

Type: string  
Required: yes

Opaque packet identifier. It MUST NOT contain patient name, patient ID, phone number, email, birthdate, address, chart number, or other PHI.

### 7.2 `created_at`

Type: RFC 3339 datetime string  
Required: yes

Timestamp when the packet was created.

### 7.3 `patient`

Type: object  
Required: yes

Allowed fields:

- `age`: integer or null
- `sex`: string or null
- `deidentified`: boolean

`patient.deidentified` MUST be `true`.

### 7.4 Narrative Fields

Fields:

- `chief_complaint`
- `clinical_notes_summary`
- `treatment_plan_summary`

Each field MUST be a string. Empty strings are allowed when source information is unavailable.

Narrative fields MUST be de-identified before packet serialization. They MUST NOT include diagnosis or treatment recommendations generated by an AI system.

### 7.5 `imaging`

Type: object  
Required: yes

Contains availability and metadata for:

- `cbct`
- `xray`
- `intraoral_scan`
- `photos`

Large imaging files MUST NOT be embedded in packet JSON. They MUST be referenced by path, URI, manifest ID, content hash, or future artifact reference.

### 7.6 `ai_ready_context`

Type: object  
Required: yes

Fields:

- `case_overview`
- `known_information`
- `missing_information`
- `clinical_review_questions`
- `llm_prompt_context`

`llm_prompt_context` MUST include non-diagnostic instructions. It MUST NOT ask a model to diagnose, recommend treatment, or replace dentist review.

### 7.7 `safety`

Type: object  
Required: yes

Fields:

- `not_for_diagnosis`
- `requires_dentist_review`
- `phi_removed`

All three fields MUST be `true` for a conforming v0.1 packet.

## 8. Privacy Requirements

Implementations MUST use allowlists when exporting clinical metadata.

Implementations MUST NOT export raw values for:

- PatientName
- PatientID
- PatientBirthDate
- PatientAddress
- InstitutionName
- InstitutionAddress
- ReferringPhysicianName
- OperatorsName
- AccessionNumber

Implementations SHOULD treat filenames as possible PHI.

Implementations SHOULD warn that pixel data, photographs, PDFs, and DICOM private tags may contain embedded identifiers.

## 11. Extension Model

Vendors MAY add extension fields under future reserved namespaces such as:

```json
{
  "extensions": {
    "vendor.example": {}
  }
}
```

Until the extension model is finalized, implementations SHOULD avoid top-level non-standard fields in strict mode.

## 12. Clinical Safety Boundary

Any consumer of the packet MUST treat it as context for clinical review only.

The packet MUST NOT be presented as:

- a diagnosis
- a treatment plan
- a clinical decision
- a replacement for professional judgment
